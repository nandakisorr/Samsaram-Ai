## Hard rules (always apply)

These baseline rules are non-negotiable for every task. They override nothing
below them — everything below is additive. This is not a child-play project;
hold an enterprise-grade bar on every diff.

**Engineering quality**

- Deliver enterprise-grade, professional solutions — no shortcuts, no
  "trickery" workarounds, no band-aid fixes that mask a root cause.
- Use TypeScript and TSX only — no `.js` / `.jsx` in application code.
- After applying new changes, always start the **production** server (build
  + run) to prove the change works outside dev mode. Never ship on dev-only
  verification.

**UI & assets**

- SVG only for icons and illustrations — **no emojis** anywhere in product UI.
- Every DOM element carries a module-specific prefixed ID
  (e.g. `{module}_{component}_{descriptor}` → `chat_send_btn`,
  `booking_date_input`). No anonymous elements.
- No unnecessary component re-renders — memoise, split, subscribe narrowly,
  or lift state. New components must stay stable under realistic prop changes.

**Architecture**

- No monolithic development. Every feature, module, or service must be
  cleanly separated with clear boundaries — own folder, own routes, own
  tests. Never grow a single file, class, or endpoint into a catch-all.

**Runtime behaviour**

- No `setInterval`, no `setTimeout` for polling, no busy-wait loops.
  Everything is **event-driven**: event bus, WebSocket push, `addEventListener`,
  promise-based `ready` signals, `AbortController`, etc.

**Observability**

- All errors — client **and** server — log to an error-log file in
  **OTEL-structured** format. Never `console.error` alone; never swallow.
- After every E2E run, read the error logs and fix anything that surfaced —
  a green test suite is **not** sufficient if the logs show errors.

**Testing (E2E)**

- Every change ships with E2E coverage run in **headed mode**.
- Every interactive surface — buttons, inputs, forms, modals, navs, menus,
  pages — has an E2E test exercising it end-to-end.
- E2E selectors always use the prefixed ID. If a target has no ID, **add
  the prefixed ID first**, then select by it — never select by text, class,
  or structural position.

In addition to text-based responses, the chatbot should generate human-like speech using a neural text-to-speech (TTS) system.
